# 项目代码检查报告与修改建议

## 1. i18n 语言切换 — 中韩混合与一致性问题

### 1.1 已发现的问题

| 位置 | 问题 | 说明 |
|------|------|------|
| **i18n.js kr** | `nav_teacher: "교사专区"` | 韩语文案中混入中文“专区”，应改为纯韩语如 `"교사 전용"` |
| **i18n.js kr** | `stroke_btn_trace: "묘홍"` | “묘홍”为“描红”音译，韩语用户更习惯“따라쓰기”，与 `home_tag3` 一致 |
| **i18n.js kr** | `home_copy: "© 2026 中文学习中心"` | 韩语下应显示韩文或英文，避免中文 |
| **hskRenderer.js** | 课次显示 `${lessonNo}과` | 中文模式下仍显示“1과”，应随语言切换为“第1课”/“1과” |
| **多处 getLang()** | 返回值不统一 | 有的返回 `ko`/`zh`（规范），有的 fallback `"kr"`；与 DICT 的 kr/cn 键、pickText(ko/kr/zh/cn) 混用可能造成个别处取不到文案 |

### 1.2 修改建议

- **统一“取当前语言键”的用法**：从数据对象（如 title、meaning）里取多语文案时，建议用 `i18n.getLangLegacy()`（返回 kr/cn/en），与 DICT 及现有数据结构一致；仅做 UI 文案时用 `i18n.getLang()` 即可。
- **修正 i18n.js**：  
  - kr：`nav_teacher` → `"교사 전용"`；`stroke_btn_trace` → `"따라쓰기"`；`home_copy` → `"© 2026 Lumina Chinese Learning Center"` 或韩文版权文案。  
  - 检查其他 kr 键是否还有中文残留。
- **hskRenderer 课次后缀**：根据 `lang` 显示“第 N 课”或“N과”，可增加 i18n key 如 `hsk_lesson_suffix_ko` / `hsk_lesson_suffix_zh`（或用现有 i18n 句式中插值）。

  - 组件/页面用带前缀的 token（如 `--panel-*`），避免与全局同名。

## 3. Router — 是否可能重复渲染

### 3.1 现有防护

- **navToken**：每次 `handleRouteChange` 自增，加载完成后若 `token !== navToken` 则不再写 DOM，避免并发导航互相覆盖。  
- **同 hash 跳过**：`!force && hash === currentHash` 且当前内容既非“加载中”也非空时，只发 `emitRouteEvent()` 不重新执行 loader/mount。  
- **启动时**：`startRouter` 中若 `!location.hash` 会设 `location.hash = _defaultHash`，再用 `queueMicrotask` 执行一次 `handleRouteChange(..., force: true)`，保证首屏一定渲染一次。

### 3.2 潜在风险与建议

- **设置 hash 触发 hashchange 的顺序**：在部分环境下，`location.hash = _defaultHash` 可能同步或异步触发 `hashchange`。若在 `addEventListener("hashchange")` 之前触发，则不会重复渲染；若在之后触发，则同 hash 会再进一次 `handleRouteChange`，此时依赖“同 hash 跳过”逻辑，一般不会重复渲染。建议在 `startRouter` 里先添加 `hashchange` 监听，再根据 `location.hash` 决定是否赋值，避免依赖实现细节。  
- **页面模块自执行 mount**：**page.hsk.js** 末尾有“Auto mount”：在脚本加载时若 `document.readyState !== "loading"` 会立即执行 `mount()`，而 router 随后也会对同一模块调用 `mod.mount(ctx)`，导致 **#hsk 被挂载两次**。建议删除 page.hsk.js 中的自执行 mount，仅由 router 调用 `mount(ctx)`。

## 4. Lessons 数据加载 — 路径问题

### 4.1 已发现的问题

| 位置 | 问题 |
|------|------|
| **hskLoader.js 第 274 行** | `const idxPath = \`/data/lessons/${version}/hsk${lv}.json\`;` 使用**绝对路径**，未考虑 `DATA_PATHS.getBase()` 或子目录部署（如 GitHub Pages 的 /repo/），fetch 会请求站根 `/data/...` 而非 `/repo/data/...`，导致 404。 |
| **hskLoader 内部** | `vocabUrl` / `lessonsUrl` / `lessonDetailUrl` 均返回以 `/` 开头的路径，同样未考虑 BASE。 |
| **dataPaths.js** | 未提供 `lessonDetailUrl`，hskLoader 只能使用自身实现的绝对路径，无法复用 DATA_PATHS 的 BASE 逻辑。 |

### 4.2 修改建议

- **dataPaths.js**：增加 `lessonDetailUrl(level, lessonNo, opts)`（或接受 `file` 参数），与 `vocabUrl`/`lessonsUrl` 一样基于 `joinPath(BASE, ...)` 和版本，供 hskLoader 优先调用。  
- **hskLoader.js**：  
  - 所有请求 lessons/vocab 的 URL 优先使用 `window.DATA_PATHS`：  
    - `lessonsUrl(lv)`、`vocabUrl(lv)`、以及新增的 `lessonDetailUrl(lv, lessonNo, opts)`。  
  - 在尝试“索引文件”时，不要使用硬编码的 `idxPath = \`/data/lessons/...\``，改为：  
    - 若 DATA_PATHS 提供 `lessonsIndexUrl(level)` 或约定 `lessonsUrl` 返回的同一目录下的 `hsk${lv}.json`，则用 DATA_PATHS 拼出带 BASE 的 URL 再 fetch；  
    - 或直接使用 `DATA_PATHS.lessonsUrl(lv)` 得到 `hsk${lv}_lessons.json`，不再单独请求 `hsk${lv}.json`，避免两套路径。  
- 若 hskLoader 内部保留 fallback 的 URL 构建，其返回值也应是“相对 BASE”的路径（例如用 `joinPath(DATA_PATHS.getBase(), "data/lessons/...")`），而不是以 `/` 开头的绝对路径。

## 5. 模块结构 — 可优化点

### 5.1 HSK 依赖未保证加载

- **现象**：`window.HSK_LOADER` 由 `hskLoader.js`（经典脚本）注册，但 **index.html 未引入该脚本**，也**没有任何地方调用 `ensureHSKDeps()`**。用户首次进入 `#hsk` 时，`page.hsk.js` 的 `loadLessons()` 会访问 `window.HSK_LOADER`，若为空则抛错。  
- **建议**：  
  - **方案 A**：在 `index.html` 中在 `dataPaths.js` 之后增加 `<script src="./ui/modules/hsk/hskLoader.js"></script>`，保证全站可用。  
  - **方案 B**：在 `#hsk` 路由的 loader 中先拉依赖再加载页面，例如：  
    `registerRoute("#hsk", () => import("./modules/hsk/hskDeps.js").then(m => m.ensureHSKDeps()).then(() => import("./pages/page.hsk.js")));`  
  - **方案 C**：在 `page.hsk.js` 的 `mount()` 开头 `await ensureHSKDeps()`，再执行后续逻辑。  
  推荐方案 A 或 B，避免首屏到 #hsk 时再串行加载脚本增加延迟。

### 5.2 语言状态两套来源

- **现象**：`core/lang.js` 提供 `getLang()`/`setLang()`，读写 `localStorage` 和 `window.APP_LANG`；`i18n.js` 也有自己的 `_lang` 与 storage。若某处只调 `lang.setLang()` 不调 `i18n.setLang()`，则界面语言与 i18n 内部状态可能不一致。  
- **建议**：以 **i18n** 为唯一真相源；`core/lang.js` 的 `setLang` 内部调用 `i18n.setLang(lang)`，`getLang` 改为直接返回 `i18n.getLang()`（或保留兼容层但注明弃用）。逐步将调用方统一为 i18n。

### 5.3 getLang 实现分散

- **现象**：`page.home.js`、`page.hsk.js`、`navBar.js`、`ui-stroke-player.js`、`dialoguePanel.js`、`wordsStep.js` 等各自实现或引用不同的 getLang/语言读取逻辑，fallback 有 `"kr"`/`"ko"` 混用。  
- **建议**：统一从 `i18n.getLang()`（规范 ko/zh/en）或 `i18n.getLangLegacy()`（kr/cn/en）读取；需要从数据对象取多语时用 getLangLegacy，与 DICT 和 pickText 的键一致，减少中韩混用和 key 取错。

### 5.4 page.hsk 自执行 mount 与 router 重复

- 见第 3 节：删除 page.hsk.js 末尾“根据 readyState 自动执行 mount()”的逻辑，只保留 `export async function mount()` 供 router 调用。

## 修改项汇总（建议实施顺序）

1. **i18n**：修正 kr 的 `nav_teacher`、`stroke_btn_trace`、`home_copy`；hskRenderer 课次后缀随语言切换。  
2. **dataPaths.js**：新增 `lessonDetailUrl`；hskLoader 内索引/详情请求全部改为使用 DATA_PATHS + BASE，去掉硬编码 `/data/...`。  
3. **router / page.hsk**：去掉 page.hsk 的 Auto mount；如需可再加固 startRouter 的 hash 设置顺序。  
4. **HSK 依赖**：在 index 中预加载 hskLoader 或在 #hsk 的 route loader 中先 `ensureHSKDeps()`。  
5. **语言统一**：lang.js 委托给 i18n；各页面/组件逐步改用 i18n.getLang() / getLangLegacy()。

## 已实施的修改（本次完成）

- **i18n**：kr 的 `nav_teacher` → "교사 전용"；`stroke_btn_trace` → "따라쓰기"；`home_copy` → "© 2026 Lumina Chinese Learning Center"。新增 `hsk_lesson_unit`（kr/cn/en），hskRenderer 课次后缀改为使用 i18n。
- **dataPaths.js**：新增 `lessonsIndexUrl(level)`、`lessonDetailUrl(level, lessonNo, opts)` 并导出。
- **hskLoader.js**：索引请求改为优先使用 `DATA_PATHS.lessonsIndexUrl(lv)`；单课详情优先使用 `DATA_PATHS.lessonDetailUrl(lv, lessonNo, { file })`；内部 fallback 的 vocabUrl/lessonsUrl/lessonDetailUrl 改为通过 `withBase()` 拼接 BASE，子目录部署时路径正确。
- **page.hsk.js**：`mount()` 开头 `await ensureHSKDeps()`，保证进入 #hsk 时 HSK_LOADER 已存在；删除末尾“Auto mount”自执行逻辑，避免与 router 重复挂载。
- **router.js**：同 hash 且已渲染完成时，即使 `force: true` 也跳过，避免 startRouter 的 queueMicrotask 与 hashchange 导致二次挂载；先注册 hashchange 再设置 `location.hash`，避免漏接事件。

以上为本次检查的结论与具体修改建议。
